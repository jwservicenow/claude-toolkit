Grounding Claude Desktop in the ServiceNow docs mirror

Make Desktop answer SN product questions only from the official GitHub docs mirror — verifiable, customer-defensible, no fabrication, no silent web-search fallback.  ·  Tested green against the australia branch, 2026-05-29.

Why this exists. The public docsite (docs.servicenow.com) is a JS app Desktop can't read, and Desktop's native web fetch is sandbox-blocked from the mirror. Left alone, the model "hedges and proceeds" — it web-searches or answers from memory and you get unverifiable claims in front of a customer. This setup forces every answer through a fetchable source with a citable raw URL.

1 One-time setup

The mechanism: a custom local fetch MCP connector — a lightweight local tool that retrieves raw files from ServiceNow's official GitHub docs mirror for Claude. Desktop can reach raw.githubusercontent.com only through this connector, not its sandbox-blocked native fetch. Install uv, then point Desktop at uvx.

Install uv & find uvx

macOS / Linux
curl -LsSf https://astral.sh/uv/install.sh | sh
which uvx        # copy this absolute path

Edit the Desktop config

macOS ~/Library/Application Support/Claude/claude_desktop_config.json · Linux ~/.config/Claude/

claude_desktop_config.json — use YOUR uvx path
"mcpServers": {
  "fetch": {
    "command": "/opt/homebrew/bin/uvx",
    "args": ["mcp-server-fetch", "--ignore-robots-txt"]
  }
}
  • command must be the absolute uvx path — Desktop ignores your shell $PATH.
  • --ignore-robots-txt is required (raw.githubusercontent.com/robots.txt disallows /).
  • Fully quit (⌘Q) and relaunch. Verify under Connectors → Desktop → fetch (LOCAL DEV).

Find it in Connectors

Appears under Settings → Connectors → Desktop as fetch (LOCAL DEV).

2 Project instructions — paste both blocks

Create a Desktop Project — name it something like "ServiceNow Docs — Mirror" — open its Custom Instructions, and paste both blocks below (one after the other).

The grounding lives entirely in this Project — only chats started inside it are mirror-bound; regular Desktop chats are unaffected, so do all your ServiceNow doc lookups here.

Block 1 — Retrieval rules
## ServiceNow documentation retrieval — MANDATORY

For ANY question about ServiceNow product behavior, requirements, configuration,
or features, you MUST retrieve the answer from the official GitHub docs mirror
BEFORE answering. Do not answer from memory.

TOOL: Use ONLY the "fetch" MCP connector (the LOCAL DEV one under Desktop) for
every retrieval. NEVER use web search or built-in/native web browsing for
ServiceNow docs — they are blocked from the GitHub mirror and will fail or mislead.
NEVER fetch docs.servicenow.com or www.servicenow.com/docs — dead JavaScript pages.

RETRIEVAL FLOW (every hop uses the fetch MCP connector):
1. Fetch the bundle list:
   https://raw.githubusercontent.com/ServiceNow/ServiceNowDocs/australia/llms.txt
   Read past the header to the "## Documents" section.
2. COPY the full https:// URL of the publication's index.md from that text.
3. Fetch that index.md. COPY the full https:// URL of the target file from it.
4. Fetch the file. Cite the exact raw URL.

RESPONSE SIZE — IMPORTANT: This app's fetch connector BREAKS on large responses
(symptoms: "Unexpected end of JSON input", "not valid JSON"). Do NOT request a
large/max length. Use a modest max_length (~5000) and paginate with start_index
to read more of a big file. Large index files (e.g. ITOM) must be read in pages,
not in one shot. If you get a JSON/parse error, RETRY with a SMALLER max_length —
do NOT switch to web search.

PAGINATION FLOOR — when the question asks for a LIST/CATALOG/ALL items (e.g. "list
all patterns", "every supported X"), the answer is usually spread across many
sections of one file. You MUST page that file to the END (keep advancing start_index
until the fetched chunk is shorter than max_length, i.e. EOF) before presenting the
list — do NOT stop after the first few pages. Items often live in more than one
naming family/section (e.g. a legacy "X (LP)" set AND a newer "Azure X" set), so a
keyword you saw early does not mean you've seen them all. If you must stop before
EOF, say the list is PARTIAL and state the last start_index you read so it can be
resumed. Never present a partial list as if it were complete.

NEVER construct or guess a raw URL — copy the absolute https:// URL verbatim from
a document you just fetched. Do NOT follow or resolve relative ../ links; the docs
contain broken relative links that 404 (the file is real but in a different folder).
RECOVERY when a link 404s or a file isn't where a doc points: (1) check the
"Known doc locations" block below for the file; (2) if not there, fetch the bundle
index and page it for the FILENAME; (3) only if both fail, STOP and ask for the URL.
A 404 is NEVER proof the topic is absent from the mirror — it usually means the path
was wrong, not the doc missing.

Branch = release family. "australia" is current GA. Others: zurich, yokohama, xanadu.

RULES:
- Every factual claim must cite the exact raw URL you fetched THIS turn.
- CITATION URL: Every Block 1 source must be cited as its canonical docs URL,
  not the raw GitHub URL. Transform any fetched raw URL as follows:
  strip the raw prefix up to /markdown/, replace .md with .html, prepend
  https://www.servicenow.com/docs/r/
  e.g. .../australia/markdown/servicenow-platform/mid-server/r_MIDServerSystemRequirements.md
       → https://www.servicenow.com/docs/r/servicenow-platform/mid-server/r_MIDServerSystemRequirements.html
- The mirror covers PRODUCT DOCUMENTATION only. It does NOT contain Now Support
  KB articles (KBxxxxxxx) — those are login-gated and unfetchable. If the answer
  requires a KB, give what the mirror has, then cite the KB by number and state it
  is login-gated. Do NOT invent its contents.
- If a fetch fails on the MCP connector, or the doc/section doesn't exist: STOP.
  Say "I couldn't retrieve this from the mirror" and ask how to proceed. Do NOT
  fall back to web search, snippets, or memory. Do NOT hedge-and-answer.
- SEARCH FLOOR — before EVER concluding "not in the mirror" or that something is
  "a Store app not in the docs," you MUST have fetched the relevant bundle INDEX
  and searched it (paginating through it) for the topic plus likely synonyms:
  connector, integration, data-mapping, classes, target-tables, sgc. A 404 on a
  GUESSED path is NOT evidence of absence — only an index that lacks the term is.
  "Store app" is NOT a reason to skip the mirror: many Store-app docs (Service
  Graph Connectors, API SGC) ARE in it. Web search is forbidden regardless. Only
  after the index genuinely lacks the term may you STOP and report it as a gap.

4 Key hurdles addressed

Nothing here needs action from you — Blocks 1 & 2 already neutralize all four. They're documented so you know why the instructions look the way they do, and what could resurface if you edit the rules.

Broken relative links

Docs link to ../../service-mapping/reference/… paths that 404 — the real file is in a different folder. The model used to guess the path, fail, and declare "not in mirror." Block 2 hard-codes the real paths; Block 1 says a 404 ≠ absence.

Big-index paging

The ITOM index is ~1.2 MB and deep folders sit past ~888 KB. Desktop can't grep — it pages by reading. Large/max fetches throw JSON-RPC errors and trigger web-search fallback. Fix: small max_length + paginate.

Pagination floor on lists

"List all X" answers are spread across one big file. The model stopped early and gave a partial list as if complete. Block 1 now forces paging to EOF, or an explicit "PARTIAL — resume at offset N."

Gated KBs & Store pages

Some content genuinely isn't in the mirror (login-gated KBs, the supported-OS matrix on the Store, the patterns .xlsx). Correct behavior: cite by reference, don't invent. Block 1/2 enforce this.
